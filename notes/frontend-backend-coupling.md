# 思想记录：前端与后端的高耦合语义

## 背景

AI 原生 admin 框架不能把前端简单理解为页面和组件渲染。

在 admin 系统中，很多前端能力都和后端能力高度相关。前端看似是 UI 问题，实际背后依赖数据模型、接口 contract、鉴权、角色权限、租户隔离、文件服务、验证码、安全策略、导入导出任务等后端语义。

如果框架只显式化前端组件，而不显式化前后端关系，AI Agent 仍然需要猜测大量上下文。

## 典型复杂点

### 1. 组件类型与组件属性

前端组件不只是视觉组件，还承载业务语义。

例如：

- 表格列来自哪些后端字段
- 表单字段对应哪个数据模型属性
- 选择器的数据源来自哪个接口
- 文件上传组件使用哪个文件服务
- 日期范围组件如何映射到查询参数
- 状态标签如何映射后端枚举

因此组件属性不能只是 UI props，还应包含业务绑定、数据来源、权限约束和验证规则。

### 2. 组件扩展

Admin 框架通常需要大量可扩展区域：

- 表格列扩展
- 表单字段扩展
- 批量操作扩展
- 工具栏扩展
- 搜索栏扩展
- 详情页区块扩展
- 导入导出扩展
- 审批/操作按钮扩展

这些扩展点必须显式注册，并说明：

- 扩展点位置
- 输入上下文
- 输出形态
- 权限要求
- 数据依赖
- 执行顺序
- 是否影响后端接口或任务

### 3. 路由、鉴权和角色权限

前端路由不能只是 URL 映射。

Admin 系统中的路由通常同时涉及：

- 菜单展示
- 页面访问权限
- 操作权限
- 数据权限
- 租户边界
- 后端 API 权限
- 审计日志

因此路由应和权限模型、菜单模型、后端接口、角色权限矩阵显式关联。

### 4. 搜索栏

搜索栏不是简单输入框集合。

它与后端查询能力高度相关：

- 哪些字段可搜索
- 支持精确匹配还是模糊搜索
- 支持哪些操作符
- 是否支持范围查询
- 是否支持枚举查询
- 是否支持跨表/关联查询
- 是否需要索引
- 是否受租户和数据权限约束

AI 原生框架应让搜索栏由后端查询 contract 或数据模型能力驱动，而不是让前端自行猜字段和参数。

### 5. 导入导出和功能栏

导入导出通常不是纯前端能力。

它涉及：

- 文件上传/下载
- 模板生成
- 字段映射
- 数据校验
- 异步任务
- 进度查询
- 错误报告
- 权限控制
- 审计记录
- 多租户数据隔离

因此功能栏按钮也应该有显式 action contract，说明按钮触发的是前端行为、后端 API、异步任务还是文件服务。

### 6. 租户管理

多租户不是某个页面功能，而是贯穿整个框架的上下文。

它影响：

- 路由访问
- 菜单展示
- 数据查询
- 表单提交
- 文件存储
- 导入导出
- 权限判断
- 审计日志
- 缓存 key
- 任务队列

AI 原生框架必须显式维护 tenant context，避免 AI 生成绕过租户隔离的代码。

### 7. 公共技能 / 公共能力

Admin 框架会反复使用一批公共能力，例如：

- 文件管理
- 图片上传
- 验证码
- 短信/邮件验证码
- 字典/枚举
- 审计日志
- 通知
- 导入导出
- 任务中心
- 操作确认
- 数据脱敏
- 国际化

这些不应该散落在项目中，而应该成为框架显式注册的 common capabilities。

每个公共能力应说明：

- 提供什么功能
- 前端如何调用
- 后端接口是什么
- 权限要求是什么
- 安全约束是什么
- 可扩展点是什么
- 错误如何处理

## 核心原则

### 前端不是孤立层

AI 原生 admin 框架中的前端应被视为后端语义的可视化和操作层。

前端组件、路由、搜索、工具栏、导入导出、租户管理、文件管理等，都应该和后端 contract、权限、数据模型和公共能力建立显式关系。

### 前后端关系必须可查询

AI Agent 应该能查询：

- 某个页面依赖哪些后端接口
- 某个按钮触发哪个 action
- 某个搜索字段映射到哪个查询参数
- 某个表格列来自哪个模型字段
- 某个导出功能对应哪个异步任务
- 某个页面需要哪些角色权限
- 某个文件上传使用哪个文件服务
- 某个验证码组件依赖哪个安全能力

### UI Schema 不应只是 UI Schema

Admin 框架中的 UI schema 应该包含业务语义：

- data binding
- permission binding
- tenant binding
- action binding
- validation binding
- backend contract binding
- audit binding

只有这样，AI 才能理解 UI 背后的真实系统行为。

## 初步规则

1. AI 原生 admin 框架必须显式维护前端对象与后端 contract 的关系。
2. 路由、菜单、页面、按钮、搜索字段、表格列、表单字段都应能绑定权限和数据来源。
3. 搜索、导入、导出、文件管理、验证码等能力应作为 common capabilities 注册，而不是散落实现。
4. 多租户上下文必须成为框架一等概念，并贯穿前端、后端、任务、文件、缓存和审计。
5. 前端扩展点必须说明输入上下文、权限要求、数据依赖和后端影响。
6. AI Agent 在修改前端功能时，必须能追踪其后端依赖和安全边界。
