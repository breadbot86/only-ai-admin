# AI 原生框架设计规则草案 v0.1

## 背景

当前主流 Web 框架和 admin 框架大多是为人类开发者设计的。它们隐含了大量经验知识，例如目录约定、路由推导、生命周期规则、组件属性、扩展点、错误处理和文档阅读方式。

这些设计对熟练的人类开发者友好，但对 AI Agent 不友好。AI 在开发时常常需要猜测框架规则、补全隐式上下文、从零散代码和文档中推断项目结构。

我们希望制定一套规则，用于指导未来 AI 原生框架的设计。

这套规则不绑定具体编程语言，也不绑定具体技术栈。它用于定义：什么样的框架才适合 AI Agent 理解、操作、验证和长期演进。

## 核心判断

一个框架是否 AI 原生，不看它是否集成了 AI 代码生成，而看它是否能被 AI 稳定地：

1. 理解
2. 查询
3. 操作
4. 验证
5. 解释
6. 演进
7. 沉淀文档和经验

换句话说，AI 原生框架不是“在传统框架里加 AI”，而是“框架本身按照 AI Agent 的工作方式重新设计”。

## 设计规则

### 1. 隐式知识必须显式化

框架可以有约定，但约定不能只存在于文档、人脑或框架魔法中。

任何框架行为都应该可以被显式查询、显式解释、显式验证。

例如：

- 文件如何映射到路由
- 组件支持哪些属性
- 某个生命周期函数何时被调用
- 某个模块如何被自动注册
- 某个扩展点的执行顺序是什么

原则：

> 隐式实现可以存在，但隐式知识不能存在。

### 2. 框架必须具备自描述能力

AI 原生框架必须能导出机器可读的框架信息，包括：

- 路由
- 模块
- 功能
- 组件
- 数据模型
- 接口
- 权限
- 扩展点
- 配置
- 文档
- 规则
- 验证状态

这些信息应该能通过 CLI、API、MCP 或其他 Agent 可调用方式访问。

### 3. 框架必须提供概念字典

框架中的核心概念必须被清晰定义。

例如：

- Module 是什么
- Feature 是什么
- Component 是什么
- Route 是什么
- Action 是什么
- Service 是什么
- Policy 是什么
- Extension Point 是什么

AI 容易混淆相近概念，因此框架需要显式说明概念之间的关系和边界。

### 4. 所有能力必须可枚举

AI Agent 应该能查询框架支持什么、不支持什么。

框架应提供能力清单，例如：

- 可创建哪些对象
- 可修改哪些对象
- 可验证哪些对象
- 可生成哪些产物
- 哪些操作是安全的
- 哪些操作需要人类确认

### 5. 扩展点必须显式注册

框架的插件、中间件、hook、生命周期、自动加载机制都应该显式注册，并且可以查询。

每个扩展点至少应说明：

- 名称
- 类型
- 输入
- 输出
- 执行顺序
- 副作用
- 所属模块
- 相关文档

### 6. 模块、功能、接口、组件都必须有 Contract

框架生成或管理的核心对象都应该有明确 contract。

Contract 至少包括：

- 目的
- 输入
- 输出
- 约束
- 副作用
- 使用示例
- 反例
- 可变更边界
- 相关测试

这使 AI 不只是知道“怎么调用”，还知道“什么时候该调用、什么时候不该调用”。

### 7. 错误必须结构化、可定位、可修复

AI 原生框架的错误信息不应该只是人类可读文本。

错误应包含：

- error code
- 简短说明
- 详细原因
- 影响对象
- 相关文件或配置
- 可能原因
- 建议修复方式
- 相关文档
- 可执行修复操作，如果安全

### 8. 框架必须内置验证能力

AI 原生框架应该提供统一验证机制，而不是依赖开发者经验。

验证对象可以包括：

- 路由一致性
- 组件属性合法性
- 接口 contract
- 模块边界
- 权限规则
- 文档完整性
- 生成文件新鲜度
- 扩展点顺序
- 数据模型兼容性

### 9. 生成物必须自描述

框架生成的不应只是代码。

生成一个模块、功能、页面、组件或接口时，应同时生成：

- 代码
- 元数据
- 文档
- 示例
- 测试或验收场景
- 修改边界说明

目标是让下一个 AI Agent 能够理解这个生成物为什么存在、如何使用、如何修改。

### 10. 文档必须进入框架生命周期

文档不是附属品，而是 AI 原生框架的一等产物。

框架应主动要求和维护：

- 模块介绍
- 功能说明
- 接口说明
- 组件说明
- 权限说明
- 设计决策
- 变更记录
- 反模式和踩坑经验

当代码、配置、规则发生变化时，框架应能提示或验证相关文档是否需要更新。

### 11. 框架必须提供 Agent API

AI 不应该只能通过直接编辑文件来使用框架。

框架应提供 Agent 可调用操作，例如：

- create_module
- create_feature
- create_route
- create_component
- register_interface
- list_capabilities
- explain_object
- validate_project
- update_documentation

这些能力可以通过 CLI、JSON-RPC、MCP、SDK 等形式暴露。

### 12. 框架必须明确代码所有权和修改边界

框架应明确说明：

- 哪些文件由框架生成
- 哪些文件由用户维护
- 哪些区域可以修改
- 哪些区域禁止修改
- 哪些修改必须通过框架命令完成
- 哪些生成物会被覆盖

这可以减少 AI 误改生成代码或破坏框架结构。

### 13. 框架必须支持 Explain

AI Agent 需要问“为什么”。

框架应能解释：

- 为什么这个文件产生这个路由
- 为什么这个组件不能这样使用
- 为什么这个接口需要权限
- 为什么这个扩展点在这个顺序执行
- 为什么验证失败
- 为什么某个生成物过期

Explain 能力是 AI 原生框架的重要组成部分。

### 14. 先协议，后语法糖

AI 原生框架应该优先设计稳定协议，而不是优先追求简短语法。

每个框架能力都应该先能被：

- 描述
- 查询
- 验证
- 解释
- 自动化操作

然后再设计人类友好的语法糖。

### 15. 框架服务长期演进，而不是一次性生成

AI 原生框架不能只关注创建项目和生成代码。

它还必须支持：

- 查询
- 修改
- 验证
- 重构
- 迁移
- 废弃
- 文档更新
- 决策沉淀
- 变更追踪

AI 原生框架的目标是支持软件系统长期演进。

### 16. 优先复用优秀生态组件，但组件选型必须显式化

AI 原生框架不应该默认从 0 开发所有能力。

对于 ORM、认证、权限、表单、表格、图表、队列、缓存、日志、监控、测试等领域，生态中已经存在很多优秀组件。框架应该优先识别、复用和组合成熟组件。

但组件选型不能只看单点能力，而应该显式记录：

- 为什么选择它
- 当前使用版本
- 支持的运行时 / 语言 / 框架版本
- 与其他组件的兼容关系
- 替代方案
- 采用后的收益和代价
- 未来是否可替换
- 它为 AI Agent 暴露哪些能力、约束和错误信息

关键原则：

> 优秀组件不是简单堆叠，AI 原生框架必须追求组件组合后的 1 + 1 > 2。

例如 ORM 不只是数据库访问层。它可能同时影响数据模型、API contract、admin 表格、表单生成、权限判断、审计日志、测试数据和文档生成。如果 ORM schema 能成为这些能力共享的语义源头，它就成为框架语义层的一部分。

因此，框架应该维护组件关系图，明确每个组件：

- 提供什么能力
- 被谁消费
- 依赖谁
- 哪些变更会产生影响
- 哪些组合规则必须验证

### 17. 前端必须显式绑定后端语义

AI 原生 admin 框架不能把前端视为孤立的页面和组件层。

在 admin 场景中，前端组件、路由、菜单、搜索栏、功能栏、导入导出、租户管理、文件管理、验证码等能力通常都与后端高度相关。

因此，框架必须显式维护前端对象与后端 contract 的关系。

例如：

- 表格列应能追踪到数据模型字段
- 表单字段应能追踪到后端校验规则
- 搜索字段应能追踪到查询参数、操作符和索引能力
- 按钮应能追踪到 action、API、权限和审计记录
- 导入导出应能追踪到文件服务、异步任务、模板、校验和错误报告
- 路由应能追踪到菜单、页面权限、角色权限、数据权限和租户边界
- 文件上传、验证码、字典、通知等公共能力应能追踪到统一服务 contract

关键原则：

> Admin 前端不是后端之外的 UI，而是后端语义的可视化和操作层。

因此，UI schema 不应只是 UI schema，它还应该包含：

- data binding
- permission binding
- tenant binding
- action binding
- validation binding
- backend contract binding
- audit binding

AI Agent 在修改前端功能时，必须能够查询这个前端对象依赖哪些后端接口、权限、租户上下文和公共能力。

### 18. 公共能力必须框架化，而不是散落实现

Admin 框架中经常出现一批公共能力，例如：

- 文件管理
- 图片上传
- 验证码
- 短信 / 邮件验证码
- 字典 / 枚举
- 审计日志
- 通知
- 导入导出
- 任务中心
- 操作确认
- 数据脱敏
- 国际化

这些能力不应由业务页面反复散落实现，而应该成为框架显式注册的 common capabilities。

每个公共能力都应说明：

- 提供什么功能
- 前端如何调用
- 后端接口是什么
- 权限要求是什么
- 安全约束是什么
- 租户边界是什么
- 可扩展点是什么
- 错误如何处理

公共能力框架化之后，AI Agent 才能在不同功能模块中稳定复用它们，而不是每次重新生成一套不一致的实现。

### 19. 代码注释必须服务 AI 理解，而不是重复代码表面含义

AI 原生框架应定义代码注释规则。

注释不应该主要用于解释语法或重复代码正在做什么，而应该沉淀 AI 难以从代码表面推断的信息，例如：

- 为什么这样设计
- 为什么不能采用更简单的实现
- 修改这里会影响哪些模块
- 这里依赖哪些权限、租户、安全或审计规则
- 这里是否是扩展点
- 这里是否是生成代码
- 这里是否禁止手动修改
- 这里对应哪份模块文档或决策记录

推荐原则：

> 注释不应解释代码正在做什么，而应解释代码为什么这样做、不能怎么改、改动会影响什么。

框架可以定义跨语言的结构化注释标签，例如：

- `@ai-purpose`
- `@ai-contract`
- `@ai-invariant`
- `@ai-extension-point`
- `@ai-generated`
- `@ai-editable`
- `@ai-do-not-edit`
- `@ai-permission`
- `@ai-tenant-boundary`
- `@ai-audit`
- `@ai-doc`
- `@ai-decision`
- `@ai-pitfall`

这些标签可由不同语言实现映射到对应注释风格。

### 20. 每个模块开发后必须沉淀模块文档

AI 原生框架中的模块不应只是代码目录。

每个模块开发后，都应沉淀模块文档，至少包含：

- 模块目的
- 模块边界
- 不负责什么
- 核心能力
- 对外接口
- 前端入口
- 后端 contract
- 权限与角色
- 租户边界
- 数据模型
- 扩展点
- 公共能力依赖
- 关键流程
- 错误处理
- 测试 / 验收场景
- 常见坑
- 相关决策记录

模块文档是 AI Agent 后续维护该模块时的长期记忆。

### 21. 文档读取引导必须配置化

AI Agent 最大的问题之一不是读不了文档，而是不知道当前任务应该读哪些文档。

因此，AI 原生框架应该维护文档读取配置，按任务类型引导 AI 加载相关文档。

例如：

- 创建模块时应读取模块 contract、文档生命周期、权限模型
- 修改前端时应读取前后端语义绑定、组件 contract、权限模型
- 添加导入导出时应读取文件管理、导入导出、任务中心、审计规则
- 修改租户相关能力时应读取租户边界、数据权限、缓存和文件隔离规则

文档读取配置可以作为框架标准产物存在，并进入验证范围。

如果某个模块有文档，但没有被配置到任何读取路径中，它对 AI 来说仍然近似不可见。

因此，框架不仅要生成文档，还要维护文档索引和读取规则。

## 初步结论

我们正在制定的是一套“AI 框架设计规则”，而不是某个具体语言框架。

它应该成为后续所有具体 admin 框架实现的上层约束和思想来源。
