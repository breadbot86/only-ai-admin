# only-ai-admin

这个仓库用于记录和沉淀「AI 原生框架设计规则」的思想。

我们当前讨论的目标不是先实现某一种编程语言的 admin 框架，而是先制定一套更上层的规则、原则和标准：什么样的框架才适合 AI Agent 理解、操作、验证和长期演进。

后续可以基于这套思想，再为不同开发语言、不同技术栈建立独立仓库，实现具体的 AI 原生 admin 框架。

## 当前定位

**only-ai-admin 是 AI 原生 admin 框架的思想与标准仓库。**

它关注：

- AI 原生框架应该遵守什么设计原则
- 框架如何把隐式知识显式化
- 框架如何面向 AI Agent 暴露自描述能力
- 框架如何沉淀模块、功能、接口、文档和决策
- 框架如何复用优秀生态组件，并让组件组合产生 1 + 1 > 2 的效果
- 框架如何显式维护前端 UI 与后端语义、权限、租户、公共能力之间的关系
- 框架如何支持长期演进，而不只是一次性代码生成

## 重要区分

本仓库不直接绑定具体语言或具体 Web 框架。

它不是：

- 某个 TypeScript/Python/Go/Rust admin 框架
- 某个前端组件库
- 某个后端脚手架
- 某个业务项目模板

它是：

- AI 原生框架设计准则
- AI 友好框架标准
- 后续语言级 admin 框架的思想源头

## 给 AI Agent 的阅读指引

如果你是 AI Agent，并且正在基于本仓库开发某个具体语言或技术栈的 `only-ai-admin-*` admin 框架，请不要只读取本 README 后就开始写代码。

你必须先把本仓库当作**上游设计标准**来读取，理解完整规则后，再设计具体语言实现。

推荐阅读顺序：

1. `standards/ai-native-framework-rules-v0.1.md`
   - 这是当前最重要的规则草案。
   - 你需要理解什么样的框架才算 AI 原生框架。
   - 具体语言实现必须尽量遵守这里的设计规则。

2. `notes/repo-positioning.md`
   - 理解本仓库与后续语言实现仓库之间的关系。
   - 本仓库制定规则，语言仓库负责实现。

3. `notes/ecosystem-component-selection.md`
   - 理解为什么具体实现不应该默认从 0 开发所有能力。
   - 在选择 ORM、权限、认证、表单、表格、任务队列、文件服务等组件时，要记录版本、兼容性、替代方案、组合关系和选型原因。

4. `notes/frontend-backend-coupling.md`
   - 理解 admin 前端不是孤立 UI，而是后端语义的可视化和操作层。
   - 组件、属性、扩展点、路由、鉴权、角色权限、搜索栏、导入导出、租户管理、文件管理、验证码等能力，都必须和后端 contract、权限、租户上下文、公共能力建立显式关系。

5. `notes/code-comments-and-doc-reading.md`
   - 理解代码注释应该如何服务 AI 理解。
   - 理解每个模块开发后为什么必须沉淀模块文档。
   - 理解为什么文档读取引导必须配置化，让 AI 根据任务类型读取正确文档。

## 给具体语言 admin 框架的实现要求

当你创建或维护某个具体语言的 admin 框架仓库时，例如：

- `only-ai-admin-ts`
- `only-ai-admin-python`
- `only-ai-admin-go`
- `only-ai-admin-java`
- `only-ai-admin-rust`

请遵循以下要求：

1. **不要只做传统脚手架。**
   - 目标不是生成一堆代码模板，而是生成 AI Agent 能持续理解、验证和演进的框架结构。

2. **必须显式化隐式规则。**
   - 路由、组件、模块、权限、扩展点、生命周期、生成文件、可编辑边界都应可查询、可解释、可验证。

3. **必须提供机器可读的自描述信息。**
   - 具体实现应考虑 manifest、schema、introspection API、CLI、MCP 或其他 Agent 可调用接口。

4. **必须把文档纳入框架生命周期。**
   - 新建模块、功能、接口、组件时，不应只生成代码，也应生成说明、contract、示例、测试或验收场景。
   - 每个模块开发后必须沉淀模块文档。
   - 文档读取引导必须配置化，让 AI 根据任务类型知道应该读取哪些文档。

5. **必须定义面向 AI 的代码注释规则。**
   - 注释不应重复代码表面含义，而应沉淀设计意图、修改边界、权限/租户/审计约束、扩展点、风险和文档链接。

6. **必须重视生态组件选型。**
   - ORM、认证、权限、表单、表格、文件、导入导出、任务队列等能力应优先评估成熟生态组件。
   - 组件不是简单堆叠，要追求组件间的语义协同和 1 + 1 > 2。

6. **必须显式维护前后端语义关系。**
   - Admin 前端中的页面、表格、表单、搜索栏、功能栏、导入导出、文件上传、验证码、租户管理等，都应能追踪到后端 contract、权限、数据模型、租户上下文、审计或公共能力。

7. **必须支持长期演进。**
   - 框架不只服务第一次生成，也要支持查询、修改、验证、解释、迁移、废弃、文档更新和决策沉淀。

## 后续仓库关系

这些仓库负责具体语言和技术栈的实现；本仓库负责标准、原则、概念和范式沉淀。
